# Project Directory Mapping

This document provides a high-level overview of the directory structure for the **yeaung** repository (Next.js App Router + Supabase).

## 1. Root Directory

```text
yeaung/
├── .agents/            # AI Agent skills, architectural guardrails, and memory logs
├── .git/               # Git repository configuration
├── frontend/           # The Next.js web application
├── slides/             # Project presentations or slide decks
├── .mcp.json           # Model Context Protocol configuration
└── README.md           # Project overview and setup instructions
```

## 2. Frontend Application (`frontend/`)

The core of the application resides entirely within the `frontend/` directory. There is no separate backend directory; backend operations (API, database communication) are handled via Next.js Server Components, Route Handlers, Server Actions, and Supabase.

### 2.1 Configuration Files
- **`package.json`**: NPM dependencies and project scripts.
- **`tailwind.config.ts`**: Tailwind CSS configuration.
- **`tsconfig.json`**: TypeScript compiler configuration.
- **`next.config.js`**: Next.js framework configuration.
- **`.env.local`**: Local environment variables (ignored by Git; must contain Supabase URLs and Keys).

### 2.2 Source Code (`frontend/src/`)

```text
frontend/src/
├── app/               # Next.js App Router directory
│   ├── api/           # Next.js Route Handlers (HTTP endpoints)
│   ├── layout.tsx     # Root application layout
│   ├── page.tsx       # Root landing page
│   └── (routes)/      # Various route groups and pages
├── components/        # Reusable UI components
│   ├── ui/            # Base UI components (e.g., Shadcn UI elements like buttons, dialogs)
│   └── shared/        # Shared application components (Layouts, Navbars)
├── hooks/             # Custom React Hooks for client-side state and side-effects
├── lib/               # Utility functions and shared libraries
│   ├── supabase/      # Supabase client initialization (browser/server clients)
│   ├── utils.ts       # Common helper functions (e.g., classname merging)
│   └── validations/   # Zod schema definitions for data validation
└── types/             # Global TypeScript type definitions and interfaces
```

## 3. Database Layer (Supabase)
While the database itself is hosted externally (or via local Supabase Docker container), interaction with the database occurs strictly via:
1. **Server Actions** (`app/actions/`) or **Route Handlers** (`app/api/`) for secure data mutation.
2. **Server Components** for secure data fetching.
3. **Supabase JS Client** configured in `lib/supabase/`.

## 4. Agent Guidelines
- When navigating the codebase, agents must assume that the Next.js `app/` directory paradigm is in use (Server Components by default).
- For database schema changes or understanding the data models, agents should refer to the TypeScript types (likely in `types/` or generated by Supabase CLI) and the validation schemas in `lib/validations/`.
